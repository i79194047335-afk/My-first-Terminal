## Правила работы с git (действуют на все фазы)

**Никогда не коммитить напрямую в основную ветку.**

### Ветка на фазу

Каждая фаза — своя ветка, от актуальной основной:

```bash
git checkout master && git pull
git checkout -b phase-1-storage      # phase-2-refactor, phase-3-lighter, ...
```

### Коммиты

- Коммитить **по логическим шагам**, не одной кучей в конце.
  Например в Фазе 1: схема БД → `core/db.py` → `core/candles.py` → тесты.
- Сообщение: что и зачем, а не «fix». Русский или английский — единообразно.
- **Не коммитить**: `*.db`, `*.db-wal`, `*.db-shm`, `data/`, `archive/`,
  `.env`, `__pycache__/`. Проверить, что они в `.gitignore` — если нет, добавить
  **до** первого коммита фазы.

### Пуш и слияние

1. Пуш ветки: `git push -u origin phase-N-name`
2. **Остановиться и показать владельцу**: что сделано, что изменилось, как проверить.
3. **Ждать явного подтверждения.** Не сливать по своей инициативе, даже если
   всё зелёное и приёмка формально пройдена.
4. Только после «да» — слияние в основную ветку и пуш.

### Откат

Пока фаза не слита, откат = смена ветки. Именно поэтому основная ветка всегда
должна оставаться рабочей: если новая архитектура сломается на живом сервере,
возврат к предыдущему состоянию должен занимать секунды, а не разбор дифов.

## Контекст

Терминал на LightweightCharts v5 + Python WS-сервер. Сейчас единственный источник —
FXCM (ForexConnect, 4 форекс-пары). Цель: мульти-провайдерный терминал с управляемым
окном истории, работой 24/7 и торговлей через API биржи Lighter.

**Состояние VPS (аудит 2026-07-12):** 26 ГБ свободно, места хватает — экономить
на хранении не требуется. Терминал занимает ~1.3 ГБ.

**Порядок фаз обязателен.** Каждая опирается на предыдущую. 24/7 невозможен, пока
рестарт теряет историю; история не переживёт рестарт, пока живёт в RAM. Поэтому
хранение — первое, а не последнее.

## Фаза 1 — Хранение и динамическое окно истории

### 1.1 Ключевое решение: ретеншен по числу баров, не по времени

Хранить **последние `KEEP_BARS = 2000` баров** на каждую пару
`(provider, symbol, tf)`. Окно скользит само, подрезается после каждой закрытой свечи.

Почему 2000, а не «2× видимой области»:
- видимая область = 122 бара (`DEFAULT_BAR_COUNT`), 2× ≈ 250;
- но `js/indicators.js` (`_tailLen`) требует хвост **300–400 баров** для EMA/RSI/MACD,
  иначе первые значения индикаторов врут;
- плюс запас на прокрутку влево.

Цена вопроса: 8 символов × 11 TF × 2000 баров × ~70 Б ≈ **12 МБ на всю базу**.
При 26 ГБ свободных обсуждать нечего.

### 1.2 Схема SQLite

```sql
PRAGMA journal_mode=WAL;

CREATE TABLE candles (
  provider TEXT, symbol TEXT, tf TEXT, time INTEGER,  -- unix СЕКУНДЫ
  o REAL, h REAL, l REAL, c REAL,
  vol_base REAL, vol_quote REAL,                      -- NULL для FXCM
  PRIMARY KEY (provider, symbol, tf, time)
) WITHOUT ROWID;

CREATE TABLE instruments (
  provider TEXT, symbol TEXT,
  price_decimals INTEGER, size_decimals INTEGER,
  min_base REAL, has_volume INTEGER,
  meta TEXT,                          -- сырой JSON провайдера
  updated INTEGER,
  PRIMARY KEY (provider, symbol)
);
```

PRIMARY KEY даёт идемпотентность: повторный бэкфил через `INSERT OR REPLACE`
не плодит дубли.

### 1.3 Подрезка окна

Вызывать после каждой **закрытой** свечи (не на каждом тике):

```sql
DELETE FROM candles
WHERE provider=? AND symbol=? AND tf=?
  AND time < (SELECT time FROM candles
              WHERE provider=? AND symbol=? AND tf=?
              ORDER BY time DESC LIMIT 1 OFFSET 2000);
```

`VACUUM` — раз в неделю, кроном. Крон-задачи по времени для ретеншена не нужны.

### 1.4 Тиковые CSV

`data/` (858 МБ, растёт ~11 МБ/сутки) — **оставить как есть**. Сырьё для будущих
бэктестов, места хватает. Сжатие и ротацию не делать: gzip сломает чтение из
существующих скриптов, а выигрыш не нужен.

### 1.5 Приёмка

- [ ] Рестарт сервера не теряет историю
- [ ] `SELECT tf, count(*) FROM candles GROUP BY tf` → ≤2000 на пару
- [ ] Фронт работает как раньше, визуально ничего не изменилось

Поведение не меняется. Только структура.

### 2.1 Выключить сигнальный контур

Это ~100 МБ записи на диск ради одной строки. Выключение флага убирает нагрузку.

### 2.3 Структура

```
terminal/
├── core/
│   ├── bus.py          # внутренний WS 127.0.0.1:8766 — фиды → хаб
│   ├── db.py           # SQLite: upsert, чтение, подрезка окна
│   └── candles.py      # нарезка тиков по TF + aggregate() (взять из server.py)
├── feeds/
│   ├── fxcm_feed.py    # py3.7 venv (forexconnect)
│   └── lighter_feed.py # py3.10 venv (websockets + raw REST)
├── common/
│   └── lighter_raw.py  # _normalize_trade, Deduper, ping/pong — ШАРИТСЯ С БОТОМ
├── hub.py              # py3.10: bus → нарезка → SQLite → WS :8765 → браузер
├── retention.json      # KEEP_BARS, белый список рынков
└── archive/
```

**Почему процессы разделены:** `forexconnect` требует Python **3.7**, SDK Lighter —
**3.8+**. В одном процессе несовместимы. На VPS есть 3.10. Бонус: падение одного
фида не роняет другой.

### 2.4 Контракт шины

```json
{"type":"tick","provider":"lighter","symbol":"BTC",
 "ts":1783869420,"price":64132.6,"size":0.0123}

{"type":"instruments","provider":"lighter","data":[
 {"symbol":"BTC","price_decimals":1,"size_decimals":5,
  "min_base":0.0002,"has_volume":true,"meta":{}}]}
```

FXCM шлёт то же с `"size": null`.

**Время везде в СЕКУНДАХ.** Lighter отдаёт миллисекунды — конвертировать на границе
фида (`ts // 1000`). Внутрь системы мс не пускать.

Хаб при нарезке: суммирует `size` → `vol_base`, `size * price` → `vol_quote`.
При `size = null` объём остаётся `null`.

### 2.5 Фронт

- Новое WS-сообщение `instruments` при коннекте → `priceFormat: {precision, minMove}`
  берётся оттуда. **Убрать захардкоженные** `precision: 5, minMove: 0.00001`
  из `createChartInternal`.
- Символы: `fxcm:EUR/USD`, `lighter:BTC`. Селектор заполняется из `instruments`,
  а не из HTML-разметки.

### 2.6 Миграция без простоя

1. `core/*` — юнит-проверка нарезки на сохранённом CSV из `data/`. Живой сервер не трогать.
2. `hub.py` поднять на порту **8767** параллельно со старым `server.py` — **через
   `git worktree`**, не переключая рабочее дерево `/root/projects/terminal`:
   ```bash
   git worktree add ../terminal-phase2 phase-2-refactor
   ```
   systemd запускает `server.py` **по пути**, а не по ветке — какой код выкачен в
   `/root/projects/terminal`, тот и исполняется. Урок Фазы 1 (`LOG.md`,
   2026-07-14): рабочее дерево осталось на feature-ветке, systemd после
   автоперезапуска подхватил чужой код, он падал на отсутствующем `_sqlite3`,
   счётчик рестартов дошёл до 2529, порт 8765 умер.
3. Проверять прод-код командой **`python3.7`**, не `python3` (3.10). На проде
   `python3.7` собран без штатного `_sqlite3` (модуль досборен вручную) — зелёные
   тесты на 3.10 ничего не говорят о том, что реально исполнится под systemd.
4. `feeds/fxcm_feed.py` — вынести стриминг. Сверить: данные в хабе идентичны старым.
5. Переключить фронт на хаб. Старый `server.py` → `archive/`.

## Фаза 3 — Lighter как источник данных

Ключи **не нужны**: рыночные данные публичны. Только mainnet.

| Что           | Как                                                              |
|---------------|------------------------------------------------------------------|
| Список рынков | `OrderApi.order_books()` → symbol, market_id, decimals, min_base  |
| Карточка      | `OrderApi.order_book_details()` → last, 24h vol, OI, funding      |
| История       | `GET /api/v1/candles`, ≤500 за вызов, без auth                    |
| Live          | WS `wss://mainnet.zklighter.elliot.ai/stream`, канал `trade/{id}` |

### Проверенные факты (не перепроверять, не додумывать)

- `resolution` принимает **только**: `1m 5m 15m 1h 4h 1d`.
  Голые числа (`"1"`, `"60"`) → 400 `invalid param`.
- `t` — **миллисекунды**.
- На WS `price`/`size` — **строки**; в REST — числа.
  Парсить `float()` **по каждому полю отдельно**, единый тип не предполагать.
- Объём: `v` = base (BTC), `V` = quote (USDC). **Хранить оба.**
- `mark_price`/`index_price` есть в JSON, но **отсутствуют в Pydantic-модели SDK** →
  ходить **raw REST**, не через SDK-модели. SDK — только для подписи ордеров (Фаза 5).
- `count_back` ведёт себя непредсказуемо — **не использовать**. Только `start`/`end`
  + пагинация по 500.
- `daily_price_change` — доля, не проценты (× 100 для UI).
- Пинг: сервер шлёт `{"type":"ping"}`, надо отвечать `{"type":"pong"}`.
  Без ответа раз в ~2 минуты соединение рвётся.

### Решения

- **Белый список рынков:** BTC, ETH, SOL, HYPE. Остальные из 223 — лениво,
  по выбору пользователя. 223 WS-подписки не открывать.
- Фид отдаёт хабу **тики (сделки), не свечи**. Хаб нарезает их той же логикой,
  что и FXCM → все TF, включая секундные, бесплатно.
- REST-свечи — **только бэкфил** при старте: 2000 баров M1 = 4 запроса по 500.
- **S5–S30 бэкфила не имеют** (минимальное разрешение API = `1m`). Копятся живьём.
  Согласовано: владельцу история нужна «начиная от M1».
- **Запинить SDK на конкретный коммит** в терминале И в боте. Сейчас в боте стоит
  `git+https://github.com/elliottech/lighter-python.git` без тега — проекты могут
  молча разъехаться по версиям.

### Переиспользовать из проекта бота

- `src/collector/list_markets.py` — копируется как есть, зависимостей нет.
- `_normalize_trade()`, `Deduper`, ping/pong из `src/collector/lighter_ticks.py` →
  вынести в **общий модуль**, шарить между ботом и терминалом. **Не дублировать.**
- `MinuteCandleBuilder` из бота **не брать** — у хаба свой нарезатель.

### Фронт

`has_volume = false`.
