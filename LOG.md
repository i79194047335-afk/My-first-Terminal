## 2026-07-14 — Фаза 1: хранение свечей в SQLite (merge `534f82e`)

**Что сделано**

- `core/db.py` — SQLite: `candles` (PK `provider, symbol, tf, time`) + `instruments`,
  upsert / чтение / подрезка окна. `KEEP_BARS = 2000` баров на каждый (symbol, tf).
- `core/candles.py` — `CandleBuilder` (нарезка тиков по ТФ) + `aggregate_higher_tf`.
- `server.py` — `db_writer` пишет закрытые свечи в отдельном потоке через очередь;
  при старте история поднимается из БД, у FXCM запрашивается **только разрыв**.
- `H1` переведён в `DIRECT_LOAD_TF` (90 дней ≈ 1560 баров): агрегация из M1 давала
  167 баров, а индикаторам нужно 300–400 хвоста.
- `tests/test_phase1.py` — 17 тестов.

**Что сработало**

Приёмка пройдена. Прогон на порту 8767 (отдельный процесс, живой сервис не тронут):
первый старт с пустой БД — полная загрузка 30 дней; рестарт — `restored from SQLite`
+ дозагрузка разрыва. Ретеншен держит потолок: ни один ТФ не превысил 2000.
В проде после рестарта — то же самое, фронт работает.

**Что НЕ сработало — и чему это научило**

1. **Прод-python3.7 был собран без `_sqlite3`.** Тесты гонялись через `python3` (3.10),
   где sqlite3 есть, поэтому 17/17 «зелёных» ничего не доказывали. На проде
   `import sqlite3` падал.
   → Модуль собран вручную из `/root/Python-3.7.17/Modules/_sqlite/` и положен в
   `lib-dynload/`; интерпретатор не пересобирался, `forexconnect` цел.
   → **Правило: код для прода проверять командой `python3.7`, а не `python3`.**

2. **Рабочее дерево git оставалось на feature-ветке.** systemd запускает
   `/root/projects/terminal/server.py` **по пути** — какая ветка выкачена, тот код и
   исполняется. Автоперезапуск systemd подхватил код ветки, тот падал на `_sqlite3`,
   счётчик рестартов дошёл до 2529, порт 8765 умер, фронт отвалился.
   → **Правило: не оставлять дерево на ветке; для параллельной работы — `git worktree`.**

**Не сделано (осталось)**

Крон-остановка снята в тот же день — см. запись ниже.

**Данные**

`market.db` — боевой файл (в `.gitignore`), окно 2000 баров/ТФ, старое удаляется.
Вечный архив — по-прежнему тиковые `data/*.csv`: из них M1 восстановим точно,

**Что сделано** (ветка `phase-2-refactor`, слита в master, `c11ca46`)

- `archive/old_data/` (249 МБ): `market_memory.json`, `signal_log.json`,
  `signal_stats.json`, `session_bias.json`, `briefing_context.json`, `History/`.
  `index_split.html` → `archive/`. `archive/` в `.gitignore`.

**Что сработало**

Рестарт чистый. RSS процесса **712 МБ → 137 МБ**. Ушли две полные перезаписи
(~100 МБ) каждые 240 секунд. Сервер к архивированным файлам не обращается:
mtime не двигался, `lsof` пуст, за 5+ минут ничего не пересоздалось.
Брифинг, молчавший с 09.07, ожил: ключ DeepSeek добавлен в `.env`, прогон
дал свежий `briefing.json` по всем 4 парам.

**Чему научило**

1. **Флаг обязан гасить и импорт, а не только вызов.** `pattern_memory` и
   `signal_tracker` читают свои JSON **на уровне модуля**: цена простого
   `import` — ~100 МБ RSS. Хуже: `pattern_memory` вешает `atexit`-сохранение,
   т.е. каждая остановка сервера = полная перезапись 73-МБ файла. Один `if`
   вокруг `process_tick` этого бы не убрал.

2. **Порядок 2.1 → 2.2 — не формальность, но и не единственная зависимость.**
   Кроме сервера те же JSON читал **крон** (`asia_watcher`, `pos_watcher`,
   `pre_session_brief`), чего в ROADMAP не было. Перед `mv` живого файла
   проверять не только `server.py`, но и `crontab -l`.

3. **Брифинг был привязан к побочному продукту сигналов.** Техническую картину
   он брал из `market_memory.json`. Выключи сигналы «в лоб» — и он бы молча
   описывал вчерашний рынок. Источник переведён на `market.db`: свечи живут
   независимо от сигнального контура.

**Известное, не чинилось**

RSS-фиды брифинга наполовину мертвы (Reuters закрыл публичные RSS): на прогоне
пришёл **1** релевантный заголовок. Календарь и техника на месте, новостная
часть худая. Не регресс Фазы 2 — так было и раньше.

Ретеншен показывает 2001–2009 баров вместо 2000: подрезка идёт раз в 200
записей (`DB_TRIM_EVERY`), а не после каждой свечи. Не дефект.
