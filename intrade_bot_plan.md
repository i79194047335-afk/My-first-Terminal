# intrade.bar bot — план реализации

## Контекст

Бинарные опционы, платформа intrade.bar. Официального API нет — сделки открываются через HTTP-запросы,
которые браузер отправляет при ручном клике по кнопкам. Бот должен перехватывать сигналы от нашего
`signal_engine.py` (Asia∩TIGHT) и автоматически открывать сделки на intrade.bar.

Счёт: демо (для начала).

## Что такое HTTP-запрос

Когда пользователь нажимает «UP» в браузере, тот отправляет на сервер intrade текстовое сообщение:

```
POST /api/trade/open
Host: intrade.bar
Cookie: session=abc123...
Content-Type: application/json

{"amount": 100, "direction": "UP", "symbol": "EURUSD", "duration": 240}
```

Сервер отвечает: `{"status": "ok", "trade_id": 12345}`.

Бот делает то же самое из Python, притворяясь браузером. Никакой магии — просто HTTP.

## Шаг 1: разведка (Иван)

1. Открыть intrade.bar в Chrome
2. F12 → вкладка **Network** (Сеть)
3. Поставить галочку «Preserve log»
4. Совершить сделку вручную: выбрать пару, сумму, время, нажать UP
5. Найти в Network запрос, связанный с открытием сделки (обычно `open`, `trade`, `order`)
6. Кликнуть на него правой кнопкой → **Copy → Copy as cURL**
7. Прислать этот cURL Клоду (следующей сессии)

**Дополнительно желательно выяснить:**
- Как долго живёт сессия? (выйти из браузера, зайти через час — разлогинило?)
- Есть ли демо-счёт? Если да — на нём и тестируем.
- Что происходит при ошибке? (намеренно ввести неверную сумму/направление — посмотреть ответ)

## Шаг 4: интеграция с signal_engine

Варианты получения сигналов (от простого к сложному):

### Вариант А: WebSocket (рекомендуемый)

Бот подключается к нашему же WebSocket-серверу (`ws://localhost:8765`) как обычный клиент,
слушает сообщения типа `signal`, при боевом сигнале отправляет запрос в intrade.

Плюс: не трогает `server.py`, живёт отдельным процессом.
Минус: нужно парсить WebSocket-сообщения.

### Вариант Б: колбэк в process_tick

В `server.py` после `_fire_signal()` добавить вызов `intrade_bot.place_trade(...)`.

Плюс: прямо в процессе тика, без задержек.
Минус: бот падает → сервер падает. Тесная связка.

### Вариант В: файловый мост

Бот читает `signal_log.json` (или отдельный `pending_trades.json`), куда сервер пишет
новые сигналы для исполнения.

Плюс: полная изоляция.
Минус: задержка на чтение файла.

**Выбор:** начать с варианта А (WebSocket), он самый безопасный.

## Шаг 5: супервизор и мониторинг

- `intrade_bot.py` → отдельный systemd-сервис `intrade-bot`
- Логгирование каждой сделки: запрос, ответ, результат
- Алерт если баланс упал ниже X или подряд N лузов
- Ежедневный отчёт: сделок открыто, PnL, WR

## Компоненты (что нужно написать)

| Файл | Назначение |
|------|-----------|
| `intrade_bot.py` | Основной цикл: WebSocket → сигнал → HTTP-запрос к intrade |
| `intrade_api.py` | Обёртка над HTTP API intrade (логин, сделка, баланс, статус) |
| `intrade_config.json` | Сессия, суммы, пары, настройки риск-менеджмента |
| `intrade_bot.log` | Лог всех действий |

## Риск-менеджмент (MVP)

- Сумма сделки: фиксированная (напр. $10 на демо)
- Не более 1 открытой сделки одновременно
- Не открывать если баланс < порога
- Пауза между сделками: минимум 30 секунд

## Архитектура

```
signal_engine.py (сервер)
       │
       │ WebSocket: {"type":"signal","direction":"UP","symbol":"EURUSD"}
       ↓
intrade_bot.py (новый процесс)
       │
       │ HTTP POST /api/trade/open
       ↓
intrade.bar
```
